# 手順書 12：実際に配信したときの動作確認

## 概要

「ライブ中バッジ」と「スケジュール（カレンダー連携）」が、**実際の配信や予定変更のときにちゃんと動くか**を、配信本番のときに自分で確認するためのチェック手順です。
これらの機能は**ローカル（自分のPCの `npm run dev`）では再現できません**。YouTube APIキーや本番のカレンダー連携が「自動ビルド（GitHub Actions）」側にしか無いためです。なので確認は**本番サイト（https://wakamatsu-iori.com ）で行います**。

---

## 大前提：いつ反映されるか（15分ルール）

- サイトは GitHub Actions が**15分ごと**に自動で作り直しています（`.github/workflows/build.yml` の `schedule: '*/15 * * * *'`）。
- つまり「配信を始めた／予定を変えた」→ サイトに出る まで**最大15分**かかります。すぐ確認したいときは下の「すぐ反映させたいとき」で手動ビルドします。

### すぐ反映させたいとき（手動ビルド）

1. GitHub のリポジトリ → 上部 **Actions** タブ
2. 左の一覧から **build-and-deploy** を選ぶ
3. 右側の **Run workflow** ボタン → `main` を選んで実行
4. 1〜2分でビルド完了（緑のチェック）→ そのあと本番サイトを再読み込み

---

## Step 1：ライブ中バッジ（配信を始めたとき）

YouTube で**ライブ配信を開始**すると、トップページ上部に赤い丸＋「配信中」＋配信タイトルのバッジが出ます（クリックでその配信へ飛ぶ）。

**確認手順：**

1. YouTube で配信を開始する（テスト配信＝限定公開でもOK。ただし**公開 or 限定公開で「ライブ中」状態**になっていること）
2. 最大15分待つ（急ぐなら上の手動ビルド）
3. 本番トップ（https://wakamatsu-iori.com ）を再読み込み
4. ✅ 上部に「● 配信中 〇〇（タイトル）」バッジが出ていれば成功
5. **配信を終了**してから、もう一度ビルドが走るのを待つ（最大15分）→ 再読み込みで**バッジが消えていれば**成功

> 仕組みのメモ：最新動画の状態が「ライブ中（`liveBroadcastContent = "live"`）」かどうかで判定します（`src/lib/youtube.ts` の `parseLiveStatus`、表示は `src/components/LiveBadge.astro`）。判定はビルド時の一発取得なので、上の「15分ルール」に従います。

---

## Step 2：スケジュール（予定の表示）

Google カレンダーの予定が、ビルドのたびに読み込まれてスケジュールに反映されます（読み取りのみ・カレンダーが唯一の正本）。

**確認手順：**

1. Google カレンダー（連携対象のもの）で予定を1件**追加・変更・削除**する
2. 最大15分待つ（急ぐなら手動ビルド）
3. 本番の `/schedule` と トップの週間ボードを再読み込み
4. ✅ 変更が反映されていれば成功（過去の予定が「休業」に化けていないかも軽く確認）

---

## Step 3：Discord から予定を入れる（`/予定` コマンド）

Discord の `/予定` コマンドで Google カレンダーに予定を**追加・削除**できます（→ 次のビルドで Step 2 のとおりサイトに反映）。詳細は `docs/setup/07-discord-calendar.md`。

**確認手順：**

1. Discord で `/予定` を実行して予定を1件追加
2. Google カレンダー側に予定が増えていれば**コマンド自体は成功**（即時）
3. あとは Step 2 と同じく、次のビルドでサイトに出るのを確認

> ⚠️ `/予定` が反応しない・拒否されるときは、許可ユーザー設定（`DISCORD_ALLOWED_USER_IDS`）に自分のDiscordユーザーIDが入っているか確認（`docs/setup/07-discord-calendar.md`）。書き込みは1日上限ガードあり（既定30件）。

---

## まだ実装していないこと（今は確認不要）

- **配信終了後に「確定URL・時刻を予定の説明欄へ自動追記する書き戻し（PATCH）」は未実装**です（構成図の Phase 2 残務。`functions/_lib/google.ts` は現状 `insert`/`delete` のみ）。
  この自動書き戻しを実装したら、この手順書に Step 4 として「配信後に予定の説明欄へURLが追記されているか」の確認を追記します。

---

## うまくいかないとき（症状 → 確認）

- **バッジが出ない** → ①本当に「ライブ中」状態か（プレミア公開・予約枠は `live` ではない）／②ビルドが走ったか（Actions の build-and-deploy が緑か）／③`YOUTUBE_API_KEY` が GitHub Secrets に入っているか（`docs/setup/05-youtube-api.md`）
- **バッジが消えない** → 配信終了後にまだビルドが走っていないだけ。最大15分 or 手動ビルドで再確認
- **予定が反映されない** → ①対象カレンダーを編集したか（連携対象は `ICS_URL` のカレンダー、`docs/setup/01-google-calendar.md`）／②ビルドが走ったか／③ビルドが**失敗**していないか（失敗時は Discord に通知が来る。前回の正常版が残るので、サイトは古いまま見える）
- **ビルドが失敗している** → Actions のログを確認。外部API取得に失敗するとビルドをわざと止めて前回デプロイを残す設計（中途半端なデータを出さないため）
