# Discord `/予定` コマンド改善 — 実装仕様（メインセッション引き継ぎ）

> 作成：母艦セッション（設計のみ）。**実装は vtuber-hp セッション**（`functions/discord/interactions.ts` 周辺）。プランモード→承認→実装→/code-review の流れで。
> 目的：日付がフリーテキスト（「6/20 または 2026/6/20」）で**書式ブレ・全角/区切りミス・年の付け忘れ（去年扱い）**が起きる → **入力を構造化＋登録前に確認**してミスを根絶する。

## 1. 方針（3本柱）
1. **日付＝フリーテキスト廃止**。`相対えらび`（選ぶだけ）または `月＋日（数値選択）`。
2. **年は自動**（入力させない）。過去日になる月日は自動で翌年。
3. **登録前に確認プレビュー**（ephemeral＋ボタン）。「確定」で初めて保存＆再ビルド。

---

## 2. コマンドのオプション定義（application command options）
| name | type | 必須 | 制約/選択肢 | 備考 |
|---|---|---|---|---|
| `relative`（相対） | STRING(choices) | 任意 | `今日/明日/明後日/今週土/今週日/来週土/来週日` | 選べば日付入力不要。最優先で解釈 |
| `month`（月） | INTEGER | 任意 | min 1 / max 12 | `relative`未指定時に使用 |
| `day`（日） | INTEGER | 任意 | min 1 / max 31 | `relative`未指定時に使用 |
| `time`（時刻） | STRING(choices) | 任意 | `19:00/20:00/21:00/21:30/22:00/その他` | 既定 `21:00`。`その他`は下記`time_custom` |
| `time_custom` | STRING | 任意 | `HH:MM`想定（正規表現検証） | `time=その他`の時だけ使用 |
| `title`（タイトル） | STRING | **必須** | 1〜60字程度 | 配信名 |
| `platform` | STRING(choices) | 任意 | `YouTube/Twitch/その他` | 既定 `YouTube` |

**入力ルール（バリデーション）**
- `relative` か（`month` と `day` の両方）の**どちらか必須**。両方無し→エラー。`month`/`day`が片方だけ→エラー。
- 存在しない日付（例 2/30, 4/31）→エラー。
- `time=その他` かつ `time_custom` が `HH:MM` でない→エラー。

---

## 3. 年の自動決定ロジック（JST基準）
すべて **JST（UTC+9）** で計算（サイト表示の`<time>`も `+09:00` で保存）。
```
now = 現在時刻(JST)
if relative 指定:
  date = now を基準に相対計算（今日=now日付 / 明日=+1 / 今週土=今週の土曜 / 来週土=翌週の土曜 …）
else:
  year = now.year
  date = year/month/day（JST）
  if date が「今日(now の日付)」より前 → year += 1 で作り直す   // 12月→1月の去年事故を防止
時刻 = time(または time_custom) を date に合成 → ISO8601(+09:00)
```
- 「今日」は過去扱いにしない（同日OK）。
- 相対で過去になるケースは無い設計（常に未来/当日を指す語のみ採用）。

---

## 4. 確認プレビュー（ephemeral＋ボタン）= 最重要
- コマンド受信→**即時 ephemeral 返信**（本人だけ表示）で解釈結果を提示：
  ```
  📅 2026/6/24（火） 21:00
  夜配信 ・ YouTube
  この内容で登録しますか？
  [✓ 確定して登録]  [✕ やり直す]
  ```
  - **必ず曜日つきで表示**（年/日付ズレに気づける）。
  - 重複（同日同時刻に既存予定）があれば `⚠ 同じ日時に既に予定があります` を添える（登録は可）。
- ボタン押下の扱い（message component interaction）：
  - **確定**：保存（§5）→ 再ビルド（§6）→ ephemeralを `✅ 登録しました：2026/6/24(火) 21:00 夜配信` に編集。
  - **やり直す**：ephemeralを `キャンセルしました。もう一度 /予定 でどうぞ` に編集（**未保存**）。
- **保留データの受け渡し**：`title` が長く `custom_id`(100字上限)に載らない可能性→**Cloudflare KV に保留エントリを一時保存**し、`custom_id="yotei:<id>"` でidだけ持つ。確定時にKV読込→コミット→KV削除（TTL 15分＝interactionトークン有効期限と揃える）。KVを使わない場合は最小情報のみcustom_idに載せ`title`は別途短縮—**KV推奨**。

---

## 5. データ形式（保存）
- 1件 = `{ "datetime": "2026-06-24T21:00:00+09:00", "title": "夜配信", "platform": "YouTube" }`。
- 既存スケジュールデータ（`src/config/…` または R2 JSON）に**追記**。実体は実装者が現行構造に合わせる。
- 表示側は `<time datetime="...">6/24(火) 21:00 JST</time>`（[制作工程/06]準拠）。過去になった枠は表示側で薄く/除外。

---

## 6. 反映（再ビルド）
- 確定時に **既存の再ビルド基盤を流用**：`repository_dispatch`（`/更新` と同じ `rebuild-site`）を発火、または Discord→R2→GitHub→Actions（[HP自動化_計画]）。新規の仕組みは作らない。

## 7. エラー文言（やさしい日本語・ephemeral）
- 日付未指定：`日付を入れてね（「相対」を選ぶか、「月」と「日」の両方を選んでね）`
- 月日片方：`「月」と「日」の両方を選んでね`
- 不存在日付：`その日付は存在しないみたい。もう一度選んでね`
- 時刻書式：`時刻は HH:MM で入れてね（例 21:00）`

---

## 8. 段階実装（推奨順）
1. **Step1（最小）**：`month`/`day` 整数＋`time` choices＋`title`/`platform`＋**確認プレビュー(確定/やり直し)**＋年自動＋JST。←ミス根絶の主効果はここで出る。
2. **Step2**：`relative` 相対えらびを追加（日常の速さ）。
3. **Step3**：重複/過去の警告、`time_custom`。

## 9. 受け入れ条件（テスト観点）
- [ ] 月13/日32等が**入力時点で弾かれる**（Discordのmin/max）。
- [ ] 今日より前の月日が**自動で翌年**になる（例：12月に「1/5」→翌年1/5）。
- [ ] 2/30 等の不存在日付でエラー文言。
- [ ] 確認「確定」で保存＆再ビルド、「やり直す」で**未保存**。
- [ ] 保存値が **JST(+09:00)** でズレない。曜日表示が正しい。
- [ ] 既存 `/更新`・スケジュール表示が壊れない。
