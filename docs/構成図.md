# VTuber公式HP v1 構成図・実装計画

YouTube主体／カレンダー源泉／承認制半自動。「放置リスクゼロの自己更新HP」を最小コストで立ち上げ、案件導線・物販導線・依存分散を取りに行く。

---

## 0. 基本アーキテクチャ（確定事項）

```
[VTuber本人]
   │ 予定を入力（普段どおり）
   ▼
[Googleカレンダー（配信予定専用・公開）] ←──┐
   │ .icsフィードを定期取得                  │ 確定URL・時刻を書き戻し（一方向・追記のみ）
   ▼                                        │
[ビルドパイプライン（GitHub Actions / Cloudflare 定期実行）]
   │  ├─ YouTube Data API：最新動画／再生リスト／upcoming・live検知
   │  ├─ BOOTH フィード：グッズ自動掲載
   │  └─ 失敗時 → Discord Webhook へアラート
   ▼
[静的サイト（Astro等）→ Cloudflare Pages / Netlify / Vercel]
   │
   └─ ライブ判定だけ「窓方式」：
      予定30分前から1分間隔で監視 → 配信終了確認まで継続（数時間で打ち切り）
      ゲリラ配信は土台の定期ビルド（15分ごと）が保険で拾う
```

- **予定の源泉はカレンダーのみ**（二重管理しない）。HP側に予定入力UIは作らない。
- **二層モデル**：計画レイヤー（カレンダー予定）→ 枠が立ったら確定レイヤーに昇格（埋め込み・カウントダウン・URL表示）。
- **書き戻しは「確定情報の追記」だけ**の非対称設計。ミラー同期はしない（競合・ループ防止）。
- **フォールバック**：ビルド失敗時は前回正常版を表示し続ける（静的サイトの性質でほぼ自動充足）＋「最終更新：◯分前」表示。

## 1. v1 に入れるもの

### 1-1. ページ構成（IA）

| ページ | 中身 | 更新方式 |
|---|---|---|
| トップ | ライブ中バッジ／次の配信カウントダウン／最新動画／お知らせ | 全自動 |
| スケジュール | 週間予定（カレンダー連動）＋確定枠の埋め込み | 全自動 |
| 動画 | 再生リスト連動で「ゲーム／歌／雑談」等に自動仕分け、切り抜き枠 | 全自動 |
| グッズ | BOOTHフィード自動掲載 → **購入はBOOTHへリンク（HPは売り場にしない）** | 全自動 |
| プロフィール / About | 設定・世界観・自己紹介 | 手動（静的） |
| お仕事依頼 | 問い合わせフォーム＋実績・メディアキット | 手動（静的）+フォーム |
| ガイドライン | 二次創作・ファンアート・切り抜きのルール | 手動（静的） |
| プライバシーポリシー | フォーム・計測で取得する情報の扱い | 手動（静的） |

回遊導線：初見 → About → 動画/配信 → フォロー（X・YouTube）→ 支援（メンバーシップ・BOOTH）

### 1-2. 自動化コア

- カレンダー読み取り（専用公開カレンダーの .ics、OAuth不要）
- YouTube Data API：最新動画・再生リスト・upcoming/live 検知（無料枠で余裕）
- ライブ判定の窓方式（予定±監視窓、終了確認まで、保険の打ち切りタイマー）
- カレンダーへの確定URL書き戻し（一方向PATCH追記）
- BOOTHフィード → グッズ欄自動反映
- **冪等性**：二重投稿・二重書き戻し防止（処理済みIDの記録）

### 1-3. 承認フロー（AI生成テキストのみ）

- 対象：週間予定まとめ告知文・お知らせ文章など「文章モノ」だけ
- 構造データ（予定表示・動画一覧・ライブバッジ）は承認不要で全自動
- 運用：AIが下書き → Notionに溜める → 本人がワンクリック承認 → 公開
- **告知文の生成プロンプトは2段に分離**：①概要欄等から事実抽出 → ②本人のX文体に変換（文体定義は設定資料として一箇所に保持）
- 承認SLA：承認されないまま配信開始 → 構造データだけ表示（文章は出さない）

### 1-4. 運用安全・監視

- **Discord Webhook 通知**：取得失敗・枠検知失敗・ビルド失敗を自分専用チャンネルへ（緊急はメンション付き、未解決リマインドあり、鳴らしすぎない）
- **緊急キルスイッチ**：自動投稿・告知生成を一括停止する手動スイッチ（炎上・訃報・休止時の「空気読めない事故」防止）
- API呼び出し上限（暴走防止）
- ドメイン自動更新ON＋支払い有効の死守（SSLはホスティング側で自動・無問題）

### 1-5. 土台（設定して放置で効く系）

- **ドメイン**：Cloudflare Registrar で取得（原価・更新値上げなし・WHOIS秘匿無料）。身バレ配慮なら .com 等＋WHOIS秘匿。**ドメイン自分名義所有＝依存分散のアンカー**
- **ホスティング**:Cloudflare Pages / Netlify / Vercel（無料枠・SSL自動・差し替え可能）
- SEO：構造化データ（人物・動画）、公式リンク明示（なりすまし対策）
- OGP：全ページ＋自動生成ページ（新着動画・配信中）もカード崩れなし
- 効果測定：GA4＋Search Console、問い合わせはGoogleフォーム→Sheets自動集計、Cookie同意は最小バナー
- スパム対策：フォームサービス＋ハニーポット（＋必要なら見えないCAPTCHA）

### 1-6. モバイルUX方針

- モバイルファースト（ファンの大半がスマホ）
- リッチ演出（流体・blur・大型アニメ）は**トップの一箇所だけ**。モバイルは軽量版/静止画にフォールバック、`prefers-reduced-motion` 対応
- 参考サイト（yui540 / akirafunakoshi / motionsites / bubbbly）は「演出の引き出し」として使う。全面採用はしない。ズーム禁止は真似しない
- 画像・サムネ最適化、表示速度を計測対象に含める

## 2. 継ぎ目だけ用意して後で足すもの

| 項目 | v1でやること（継ぎ目） | 後で足すもの |
|---|---|---|
| X自動投稿 | 「告知文生成」と「出力先」を分離した設計にしておく | X出力アダプタ＋承認必須＋投稿上限＋書き込み課金管理 |
| マルチ配信先 | 配信データを「プラットフォーム名＋URL＋状態」の汎用形に。YouTubeを最初のアダプタ扱い | Twitch・ツイキャス等のアダプタ追加、ライブ判定統合 |
| ファン直結チャネル | フッターにRSS。メール登録枠のプレースホルダ | ニュースレター／Web Push の配信開始リマインダー |
| 複数人運営 | 承認フローを「承認者」概念で抽象化しておく | マネージャー権限・編集権限の分離 |
| 多言語 | テキストを構造化して持つ（直書きしない） | 英語併記（翻訳は承認制） |

X埋め込み：Xの中身は公式埋め込みタイムラインで表示するだけ（API読み取り課金を回避）。加工が必要になったら自分の投稿のみ少量従量課金。

## 3. v1では後回し（必要になったら）

- 専用承認画面（まずNotionで回す。回らなくなったら作る）
- ライブ判定の完全リアルタイム化（窓方式＋15分ビルドで実用十分）
- HP内お知らせのX自動ポスト連携（→継ぎ目から着手）
- ボイス販売・ブログ等の配信外コンテンツページ
- マイルストーン自動検知（登録者記念等）
- 受信モデレーション強化（まずはフォーム＋目視で開始）
- ファンアート掲載（許諾フロー設計が先）
- GA4→Sheetsダッシュボード化

## 4. 法務・身バレまわりの方針（要点）

- **HPは売り場にしない**：決済・配送・特商法表記・購入者個人情報はBOOTH側に乗せる。HPはショーケース＋リンクのみ → 自ドメインに特商法ページ不要
- BOOTH上も、「販売業者」に該当しない規模なら住所等の常時掲載は不要とされる（目安：月200点以上出品／月100万円・年1000万円超など）。規模が育ったらバーチャルオフィス住所＋匿名配送で対応
- WHOIS秘匿（Cloudflare無料）で登録者情報も保護
- ※法務の最終判断は専門家確認を前提とする（特に物販規模が育った時点で一度）

## 5. 作る順番（実装ロードマップ）

**Phase 1：土台（最初の週末レベル）**
1. ドメイン取得（Cloudflare）＋ホスティング接続＋SSL確認
2. 静的サイト雛形（Astro等）＋ページ骨格（IA通り）
3. 配信予定専用カレンダー作成 → .ics読み取り → 週間スケジュール表示
4. 定期ビルド（15分）＋失敗時Discord通知

**Phase 2：自動更新コア**
5. YouTube API：最新動画＋再生リスト自動分類
6. ライブ判定の窓方式＋確定枠の埋め込み＋カレンダー書き戻し
7. BOOTHフィード→グッズ欄
8. フォールバック表示（最終更新時刻）＋冪等性

**Phase 3：HPの存在意義**
9. お仕事依頼ページ（フォーム＋スパム対策＋実績/メディアキット）
10. ガイドライン・プロフィール・プライバシーポリシー
11. SEO/OGP/構造化データ、GA4/Search Console

**Phase 4：半自動の頭脳**
12. 告知文生成（概要欄抽出→文体変換の2段プロンプト）→Notion承認キュー
13. 緊急キルスイッチ
14. デザイン作り込み（参考サイトの演出を絞って導入、モバイル間引き）

**Phase 5（任意・後日）**
15. X自動投稿アダプタ／ニュースレター／多配信対応／専用承認画面

## 7. このHPが満たす当初目的との対応

| 当初の論点 | この設計での答え |
|---|---|
| 放置リスク（最大のやらない理由） | カレンダー源泉＋自動更新＋故障通知で消滅 |
| 案件導線（HP最大の価値） | お仕事依頼ページ＋メディアキット（Phase 3） |
| なりすまし対策 | 公式ドメイン＋構造化データ＋公式リンク明示 |
| プラットフォーム依存 | 自分名義ドメイン＋差し替え可能ホスト＋（後日）オウンド連絡手段 |
| 身バレ | WHOIS秘匿＋BOOTHに特商法を乗せる＋匿名配送 |
| 空気読めない自動化事故 | 承認制（文章のみ）＋キルスイッチ |
